PROJECT OVERVIEW - CAPITAL FARM BACKEND

Purpose
- Backend API for a farming/production game where players create accounts, manage sessions, build wealth, manage land, plant crops, grow them over time, harvest resources, and progress through an expanding production chain.

Current Tech Stack
- Python + FastAPI
- SQLAlchemy ORM + SQLite
- JWT auth (python-jose)
- Password hashing with passlib/bcrypt

What Is Already Implemented
- Player registration with username + email
- Player login with access token + refresh token flow
- Token refresh and logout with token/session revocation
- Protected routes via JWT validation
- Player profile module with metadata and stats container
- Wealth progression system based on player assets
- Wallet deposit endpoint and wallet history endpoint
- Unified player storage with item catalog, quantities, categories, and capacity limit
- Land grid bootstrap with coordinate, occupancy, square-tier expansion, and weekly tax rules
- Crop type catalog for the current stage (`wheat`, `corn`, `soy`)
- Planting endpoint with crop-specific seed consumption and plot occupancy validation
- Time-based crop growth state machine (`planted`, `growing`, `ready`)
- Crop readiness endpoint and player crop listing
- Harvest endpoint with storage reward integration and plot release

API Behavior (current)
- /register creates player, profile, stats, base storage, and default land setup atomically
- /login returns bearer access token and refresh token
- /token/refresh rotates refresh session and returns new tokens
- /logout revokes active session and access token
- /me returns authenticated player account data
- /profile/me returns authenticated player profile and stats
- /progression/me returns current wealth XP, max wealth XP, level, unlocks, and wealth breakdown
- /wallet/deposit updates player balance
- /wallet/history returns player-scoped financial ledger
- /storage/me returns the player storage grouped by category
- /inventory/me is a compatibility alias that returns the same storage view
- /inventory/items/add currently adds directly to storage and is still a development utility endpoint
- /land/me returns the player land grid with width, height, farm size, next expansion context, tax context, occupied plot count, and embedded crop snapshots
- /land/plots creates extra plots while preventing duplicate coordinates and invalid manual planted state
- /land/expand upgrades the farm to the next square tier, deducts expansion cost, records an expense transaction, and returns the updated grid
- /land/plots/{plot_id}/state allows only safe manual soil-state changes on non-occupied plots
- /crops/types returns available crop catalog for this game stage
- /crops/plant validates land + crop-specific seed and persists planting
- /crops/me returns player-owned crop records with growth metrics
- /crops/{crop_id}/state returns the synchronized crop lifecycle state
- /crops/{crop_id}/harvest validates readiness, rewards output into storage, and frees the plot

Crop Lifecycle Behavior (current)
- Planting stores the crop on a specific `land_plot_id`
- Growth is deterministic and derived from `planted_at` + `growth_time_seconds`
- States transition logically across `planted`, `growing`, and `ready`
- Harvest returns the configured crop output to storage
- Harvested crops cannot be harvested again
- Harvest releases the associated land plot back to `empty`

Land Grid Rules (current)
- Plot coordinates are unique per player
- Farm starts at `3x3` and expands by full square tiers only: `4x4`, `5x5`, `6x6`, and so on
- Expansion price grows exponentially by the current farm tier
- Weekly land tax is based on the number of extra plots beyond the starter farm
- Existing irregular farms are normalized by the next square-tier expansion instead of keeping isolated plot progression
- Occupancy is enforced during planting
- `planted` is reserved for crop actions and cannot be set manually through land editing endpoints
- Occupied plots cannot be manually changed until the crop is harvested
- `/land/me` is the main farm-rendering endpoint for the client because each occupied plot can include its current crop snapshot and land economy context

Land Economy Rules (current)
- Expansion cost is an economic sink separate from weekly tax
- Weekly tax is accrued lazily from `last_land_tax_at` instead of requiring a background worker
- Due tax is applied when the player hits land/progression-sensitive flows
- Tax charges are recorded as wallet `expense` entries and affect balance/wealth progression

Wealth Progression Rules (current)
- `wealth_xp` represents the player's current wealth and can go down
- `max_wealth_xp` stores the highest wealth ever achieved
- `level` is derived from `max_wealth_xp`, so unlocked level milestones are preserved
- Wealth currently includes:
  - wallet balance
  - items in storage
  - value of active planted crops

Project Direction (from planning)
- Remaining pre-Unity backend work should focus on farm expansion, pricing/economy rules, transaction safety, and API stability
- Animal systems, market dynamics, multiplayer/social, analytics, and Unity integration remain later milestones
- Development utilities should gradually be replaced by final product actions as the game loop becomes fully server-authoritative

Operational Notes for Codex
- Use venv at ./venv
- Start server with: python -m uvicorn app.main:app --reload
- Keep SECRET_KEY configured by environment when possible
- Prefer transaction-safe writes for player economy, storage, land, crop, and harvest operations
- Treat `/land/me` as the primary farm snapshot endpoint for client rendering
